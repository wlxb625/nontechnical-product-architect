# 地区可用性、支付与上线可行性指南

技术上能实现，不代表用户实际上能使用。为非技术用户推荐任何云服务、AI API、数据库、部署平台、短信、邮件或支付服务前，必须先完成地区可行性检查。

## 一、先确认用户的现实条件

至少从用户已有信息中提取或明确标记：

- 用户所在国家或地区；
- 产品主要面向的用户地区；
- 是否拥有可进行境外线上支付的信用卡或借记卡；
- 是否只能使用支付宝、微信支付、银联储蓄卡或国内对公付款；
- 是否能接受英文控制台和英文客服；
- 是否需要中国大陆稳定访问；
- 是否准备使用中国大陆服务器和自定义域名；
- 是否有个人或企业实名认证条件；
- 是否准备办理 ICP 备案或其他上线手续；
- 是否需要国内短信、微信登录、支付宝或微信支付。

如果这些信息会改变技术选择，不得直接采用默认海外技术栈。

用户已经亲自试用某个平台，并明确说明免费开户、创建项目或部署时被要求绑定支付卡，应将其视为该用户当前账户和地区下的已确认事实。不要用泛化的“官方有免费版”反驳用户的实际使用结果。

## 二、地区可行性是硬门槛

每个第三方服务都必须回答以下问题：

### 1. 它是什么

不能只写服务名称。要说明它属于数据库、身份认证、部署平台、对象存储、AI API，还是多个能力的组合。

### 2. 它在当前产品中负责什么

说明它具体替代了哪些需要自行开发或运维的部分。

### 3. 用户能不能注册和访问

检查：

- 用户所在地区是否允许注册；
- 是否要求手机号、实名认证、企业认证或特定地区地址；
- 控制台和 API 在目标地区是否能够稳定访问；
- 面向最终用户时是否存在明显网络延迟或不可用风险。

### 4. 免费版是否真的可以开始使用

不能只确认页面上写着“Free”或“免费额度”。必须继续检查：

- 注册免费账户是否要求先绑定银行卡；
- 创建第一个项目、数据库或部署是否要求添加付款方式；
- 是否需要通过小额预授权或支付验证；
- 是否只接受能够处理境外线上交易的信用卡或借记卡；
- 没有付款方式时，免费功能是否仍可实际启用；
- 免费额度结束后是否自动扣费，还是直接停服；
- 用户当前的银行卡是否真的能够完成该平台验证。

“有免费套餐”和“无需支付卡即可使用”不是一回事。

### 5. 用户能不能持续付款

必须写明：

- 免费版能否完成原型；
- 正式上线是否必须升级付费；
- 支持信用卡、借记卡、银联、支付宝、微信支付、账户余额还是对公付款；
- 结算币种；
- 是否存在自动续费、按量超额、欠费停服或数据删除风险；
- 用户当前是否具备对应付款方式。

不要因为免费版暂时能打开，就把它描述成适合长期上线。

### 6. 怎么使用

至少给出非技术用户能够理解的过程：

> 注册账号 → 完成身份或付款验证 → 创建项目或环境 → 配置身份认证/数据库/部署 → 添加环境变量 → 部署测试 → 绑定域名 → 设置费用上限和告警。

不需要展开成完整教程，但必须让用户知道该服务不是“选中名称后自动完成”。

### 7. 在当前项目中是否推荐

使用以下状态之一：

- **推荐使用**：注册、免费启用、付款、访问、维护和上线条件都与用户匹配；
- **仅适合免费原型**：能够实际启用并测试，但付费、网络、合规或长期稳定性不适合正式上线；
- **不适合当前用户**：连免费启用都要求用户不具备的付款方式，或地区访问、实名认证、维护条件不匹配；
- **需要用户先确认**：缺少关键现实信息；
- **当前无法核实**：官方资料不明确或近期发生变化。

如果用户连免费计划的付款验证都无法完成，不得标记为“仅适合免费原型”，应标记为“不适合当前用户”。

## 三、中国大陆个人开发者的默认规则

当用户位于中国大陆，明确没有境外支付卡，或主要面向中国大陆用户时：

1. 不默认推荐只能使用境外信用卡持续付费的服务作为生产环境核心依赖。
2. 如果免费账户或免费部署也要求绑定可进行境外支付的银行卡，应直接标记为“不适合当前用户”。
3. 海外服务只有在无需用户不具备的付款验证、且确实可以启用时，才可标记为“仅适合免费原型”。
4. 优先比较能够使用人民币、本地云账户余额或国内常用支付渠道的方案。
5. 优先考虑中国大陆访问稳定性、实名认证、域名备案、短信签名、内容安全和数据合规。
6. 不要把“页面写着免费”或“可以注册账号”等同于“用户可以实际创建项目”。
7. 国内服务也不能只报名字，仍需解释产品能力、费用结构、限制和部署流程。

## 四、常见服务示例

以下内容用于说明输出方式。具体价格、免费额度、付款支持、是否要求绑卡和产品能力可能变化，实际推荐时必须重新查看官方资料，并优先采用用户的实际试用结果。

### Supabase

**它是什么**  
一个托管后端平台，主要提供 PostgreSQL 数据库、用户认证、文件存储、实时能力和服务端函数。

**在产品中负责什么**  
可以代替个人开发者自行搭建数据库、登录系统和部分后端基础设施。

**现实可用性检查**  
不能仅因为存在免费项目就判断用户可以使用。应检查当前账户、地区和注册流程是否要求添加能够进行境外线上支付的银行卡，才能创建项目、完成验证或启用相关能力。正式产品还需要考虑美元结算、配额、项目暂停和超额使用。

**怎么用**  
注册账号，按要求完成账户验证，创建项目，获得项目 URL 和密钥，建立数据表与权限策略，把环境变量配置到应用服务端，再接入登录、数据库和存储 SDK。

**对中国大陆无境外支付卡用户的默认结论**  
如果用户实际尝试后确认免费使用也要求绑定其无法提供的支付卡，应标记为“不适合当前用户”，而不是“可用于免费原型”。

### Vercel

**它是什么**  
一个前端和全栈 Web 应用部署平台，尤其常用于部署 Next.js 项目。

**在产品中负责什么**  
连接 Git 仓库、自动构建代码、生成线上网址、管理环境变量和部署版本。

**现实可用性检查**  
不能只查看是否存在 Hobby 或免费计划。应检查当前注册、团队、部署或功能启用流程是否要求添加付款方式或完成银行卡验证。正式项目还需要评估商业使用、资源配额、中国大陆访问质量和美元付费升级。

**怎么用**  
连接 GitHub 仓库，按要求完成账户验证，导入项目，设置构建命令与环境变量，执行部署，测试成功后绑定域名。

**对中国大陆无境外支付卡用户的默认结论**  
如果用户实际遇到免费使用也必须绑定其无法使用的境外支付卡，应标记为“不适合当前用户”，不能继续作为默认原型部署方案。

### 腾讯云开发 CloudBase

**它是什么**  
腾讯云提供的全栈云开发平台，包含身份认证、数据库、云函数或云托管、文件存储和 Web 应用托管等能力。

**在产品中负责什么**  
可以同时承担部分 Supabase 的后端职责和部分 Vercel 的托管部署职责。

**现实可用性检查**  
更适合需要中国大陆访问、人民币云账单和国内账号体系的用户。仍需核对具体套餐、实名认证、按量付费、免费体验期限、是否需要账户余额、域名和备案要求。

**怎么用**  
注册并实名认证腾讯云账号，创建 CloudBase 环境，选择数据库和登录方式，配置云函数或云托管，部署 Web 前端，设置安全规则、费用告警和域名。

**默认结论**  
当用户主要在中国大陆、没有境外支付卡，并需要快速完成 Web 或小程序 MVP 时，可作为优先比较方案，但不是无条件唯一答案。

## 五、国内方案比较时不要只换品牌

“把 Supabase 换成国内数据库、把 Vercel 换成国内服务器”仍然可能让非技术用户面对多个复杂控制台。

优先比较两类路线：

### 一体化云开发路线

例如带有数据库、认证、云函数、存储和托管的国内云开发平台。

适合：

- 非技术个人开发者；
- 快速 MVP；
- 希望减少服务器运维；
- 小程序或 Web 应用；
- 能接受一定平台绑定。

### 自建后端路线

例如国内云服务器或容器平台，加上 MySQL/PostgreSQL、对象存储和自行实现的身份认证。

适合：

- 需要更强控制权；
- 已有开发和运维能力；
- 有明确迁移或合规要求；
- 能承担更高部署、备份和安全成本。

Skill 必须说明两条路线的差异，不能因为国内服务可付款就自动认为它更简单。
