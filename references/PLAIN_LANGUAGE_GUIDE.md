# 通俗解释指南

## 目标

解释不是附加的百科知识，而是帮助非技术用户判断当前产品方案。

## 解释顺序

1. 先说它解决什么问题。
2. 再用生活类比建立直觉。
3. 再给专业名称。
4. 说明它在当前产品中的具体位置。
5. 说明不使用、替换或出错的后果。
6. 只讲到当前决策所需的深度。

## 推荐句式

- “你可以先把它理解为……”
- “在这个产品里，它主要负责……”
- “第一版需要它，是因为……”
- “现在不需要深入到……，只要知道……”
- “这不是唯一方案，另一种做法是……”
- “选择 A 会更简单，但代价是……”

## 避免的方式

- 用另一个未解释的术语解释术语。
- 一上来给出框架源码、SQL 或复杂架构图。
- 把技术比喻当成准确技术定义，不补充边界。
- 每个词都写成百科全书，淹没产品结论。
- 用“显然”“很简单”“你只需要”暗示用户不该困惑。

## 示例：API

**一句话理解**

API 是两个程序按照约定互相传递请求和结果的方式。

**类比**

像餐厅里顾客通过服务员向厨房下单。顾客不用进入厨房，厨房也只接受规定格式的订单。

**在当前产品中的作用**

网站后台通过模型 API 把用户的问题发送给 AI 服务，再把回复交给页面显示。

**为什么不能直接放在网页里**

模型密钥放在浏览器中容易被别人看到和盗用，因此通常由后台代为调用。

**当前理解深度**

用户暂时只需知道 API 是系统之间的沟通接口，以及调用可能产生费用。

## 示例：数据库

**一句话理解**

数据库是产品长期、可查询地保存信息的系统。

**类比**

它不像把所有资料扔进一个文件夹，而是像有规则的多张表格，知道用户、任务、消息之间如何关联。

**当前产品中的作用**

保存账号、目标、任务进度、聊天记录和 AI 记忆。

**重要判断**

需要区分哪些内容必须保存、保存多久、谁能读取、用户能否删除。
